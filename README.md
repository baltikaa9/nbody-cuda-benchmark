# Отчет: Оптимизация CUDA N-Body (GTX 1650 SUPER)

## 1. Сравнение памяти (Global vs Shared)
Использование разделяемой памяти (Shared Memory) значительно повышает эффективность за счет исключения повторных запросов к глобальной памяти (VRAM).

* **Максимальный эффект**: Наибольший прирост наблюдается при малых объемах данных ($N = 4096$). Например, для конфигурации `float3` и `BS = 128` производительность увеличивается с **1.077 TFLOPS** (Global) до **1.938 TFLOPS** (Shared) — прирост составляет **почти 80% (1.8x)**.
* **Поведение на больших N**: С ростом количества тел ($N = 65536$) разрыв сокращается, так как задержки памяти маскируются высокой плотностью вычислений. Тем не менее Shared-версия сохраняет стабильное преимущество в ~8% (**2.477 TFLOPS** против **2.291 TFLOPS** при `BS = 128`).

![Сравнение памяти](plot_comparison.png)

## 2. Влияние размера блока (Block Size)
Размер блока (BS) критически влияет на утилизацию вычислительных блоков (Occupancy) графического процессора.

* **Оптимум**: Лучшие результаты достигаются при значениях **BS = 128** или **BS = 256**.
* **Падение на BS = 512 и 1024**: 
  * При малом числе тел ($N = 4096$) увеличение размера блока до 1024 приводит к катастрофическому падению производительности Shared-версии — с **1.974 TFLOPS** (на `BS=256`) до **0.590 TFLOPS** (на `BS=1024`). Это падение эффективности **в 3.34 раза**.
  * **Причина**: Большие блоки требуют большего количества регистров и Shared-памяти на один мультипроцессор (SM), из-за чего планировщик CUDA не может запустить достаточное количество параллельных варпов (Warp) для скрытия задержек инструкций.

## 3. Производительность типов данных (FP32 vs FP64)
Тест наглядно демонстрирует аппаратное ограничение потребительских видеокарт GeForce (микроархитектура Turing).

* **FP32 (`float3` / `float4`)**: Показывает высокую эффективность. Абсолютный пик производительности всей системы достигается на конфигурации **`shared, float3, N = 32768, BS = 128`** и составляет **2.644 TFLOPS**. Версия `float4` идет очень близко, показывая максимум в **2.549 TFLOPS**.
* **FP64 (`double3` / `double4`)**: Производительность падает до критических значений, не превышая **0.070 TFLOPS** (70 GFLOPS) во всех тестах.
* **Вывод**: Вычисления с двойной точностью (Double) выполняются в **37.77 раз медленнее**, чем с одинарной (Float). Это точно отражает аппаратную специфику GTX 1650 SUPER, где ядра FP64 урезаны в целях экономии площади кристалла.

## 4. Анализ задержек (Latency) и масштабирования
Эффективность расчета одной пары тел (`ms_per_pair`) улучшается по мере увеличения размера задачи $N$, что указывает на полное насыщение GPU работой.

* Для лучшей конфигурации (`shared, float3, BS = 128`) время на расчет одной пары снижается с **$9.286 \times 10^{-9}$ мс** ($N = 4096$) до минимальных **$6.808 \times 10^{-9}$ мс** при достижении пика на $N = 32768$.
* На $N = 65536$ накладные расходы немного возрастают (время пары увеличивается до $7.266 \times 10^{-9}$ мс), слегка снижая общую производительность до 2.477 TFLOPS.

### Итоговая рекомендация
Для получения максимальной производительности на данном оборудовании конфигурация по умолчанию должна использовать: 
**Shared Memory + векторный тип `float3` + Block Size 128 (или 256) + объем выборки $N = 32768$**.
